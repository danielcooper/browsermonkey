Introduction

This document describes the requirements specification for the Software Engineering course project by Software House One. The goal of the project, “BrowserMonkey” is to develop a new application browser for documents written in Hypertext Mark-up Language (HTML).

 

The browser is intended for use in agricultural machinery, for example a tractor dashboard, to display documents such as the machinery user manual. It will read and display a document written in HTML, including extensions added by Netscape, although not all HTML and Netscape extensions functionality will be implemented.  It was also agreed to provide some features in addition to those specified in the Customer Requirements document1. This document describes the analysis and breakdown of the full customer requirements as understood from the Customer Requirements document1 and the customer meeting in week five. It shall be used as a basis for the acceptance criteria and high-level design specification.

 

 

1      Requirement 1 – Basic Renderer

A basic text renderer, including a simple parser with internet files capability, renderer and Graphical User Interface (GUI).

1.1    Overview

The document character set will be the ASCII character set listed in Appendix B.

The list of HTML standard character entities (such as &amp;) which BrowserMonkey supports is given in Appendix C.  Supported character entities can be referenced by their names, or by their decimal / hexadecimal equivalents, for example:

 

     this is a pound &pound; so is this &#163; so is this &#xA3;

 

will render as:

 

this is a pound £ so is this £ so is this £

 

 

Entity names are case sensitive, and unrecognised entities will be rendered as normal text, for example:

 

some &quot;text &Quot; &amp; and this is some more &foo; text

 

will render as:

 

some "text &Quot; & and this is some more &foo; text

 

Hexadecimal entity references are not case sensitive, so &#xa3; &#xA3; &#Xa3; and &#XA3; will all render the same character entity.


1.2    Whitespace

Spaces inside tag brackets, e.g. <         b      > will be ignored. Spaces within text will be rendered but multiple space characters will render as only 1 space character, for example:

 

         some     <b>     text and some         more text

 

will render as:

 

         some text and some more text

 

Whitespace will be ignored inside and outside tag brackets. A list of the ASCII characters considered to be whitespace is included in appendix B.

 

Carriage return (CR), linefeed (LF) and carriage return + linefeed (CRLF) will each be treated as a single space character. Multiple instances will render as a single space.

 

Spaces and whitespace characters are handled differently when enclosed by <pre> </pre> tags.  For further explanation see the preformatted text description in section 5.9.

1.3    Starting the program

The program will be started from the command line:

 

java BrowserMonkey

 

If no filename or address is passed as an argument, BrowserMonkey will attempt to open “index.html” from the current local directory, and if “index.html” does not exist it will attempt to open “index.htm”. If “index.htm” also does not exist, BrowserMonkey will start with no document open, and will display an error page instead.

 

 

A default index.html file will be supplied with the BrowserMonkey source code.

1.4    Opening an HTML document

BrowserMonkey will open documents which are held locally, or accessed via the internet. Any file extension is permitted, although all files will be rendered as HTML documents, regardless of extension.

1.4.1    Opening a document from the command line

A document can be opened by specifying its path, filename and extension as an argument when starting the program from the command line. For example to start BrowserMonkey with a locally held file:

 

         java BrowserMonkey C:\DataFiles\Manuals\thresher.html

 

To start BrowserMonkey with a file accessed via the internet:

 

         java BrowserMonkey www.customersite.com/thresher.html

 

If the file is not found, an error page will be displayed. If the file extension is omitted, BrowserMonkey will assume that only a file path has been entered, and will attempt to open “index.html” from the specified path, for example:

 

         java BrowserMonkey C:\DataFiles\Manuals

 

will cause the program to try opening C:\DataFiles\Manuals\index.html. If “index.html” does not exist it will attempt to open “index.htm” from the same path. If “index.htm” also does not exist, BrowserMonkey will start with no document open, and will display an error page instead.

 

For an internet file path, the IP address may be specified instead of the domain name, for example:

 

         java BrowserMonkey 192.168.10.0/thresher.html

 

 

1.4.2    Opening a document from the address bar

When the browser is already running, a file can be opened by typing its path, filename and extension directly into the address bar and clicking the “Go” button. If the file is to be accessed via the internet, the address must be prefixed with “http://”.

 

As with opening a document from the command line, if the file is not found, an error page will be displayed. If the file extension is omitted, BrowserMonkey will assume that only a file path has been entered, and will attempt to open “index.html” and then “index.htm” from the specified path. If the file cannot be found, the error page will be displayed.

 

Also, as with opening a document from the command line, the IP address may be specified instead of the domain name.

 

The path, filename and extension (including “http://” if entered) are limited to a maximum of 255 characters.

 

1.4.3    Opening a document via the browse dialogue

Alternatively, the user can click the “Browse” button to open a dialogue box which allows the user to navigate to the desired locally held file.

 

 

1.5    Graphical User Interface

A prototype of the BrowserMonkey GUI can be seen in Appendix E.

1.5.1    Window title

Text within the <title> </title> tags will be displayed as the window title. For example,

 

<title> Example HTML Page Title </title>

 

will be displayed as:

 

 

Whitespace in the text within the title tags will be ignored. Spaces within the text will be displayed including multiple space characters. HTML character entities will also be displayed. Any tags nested within the title tags will be displayed as part of the title text, for example:

 

<title> hello <b> w</b>orld          how are &#quot; you</title>

 

will be displayed as:

 

         hello <b> w</b>orld          how are “ you

 

If the document does not contain a title element, the window title will be set to “BrowserMonkey – untitled document”. If no document is open, the window title will be set to “BrowserMonkey – No open document”.

 

1.5.2    Scrolling

A vertical scroll bar will provide the ability to view a page which is too long to fit within the visible window. If the page fits within the window, the vertical scroll bar will be greyed-out.

A horizontal scroll bar will only appear if the page contains non-breaking text which is wider than the window width.  

 

1.5.3    Altering window size and position

The BrowserMonkey window can be moved and resized. Elements will resize inside the window as far as is practicable for them to do so. Tables, for example, can only be resized down so far, before they become unrecognisable as tables. If the window is resized such that it becomes narrower than any non-breaking text, the horizontal scroll bar will appear, and if the window is resized to become wider than any non-breaking text, the horizontal scroll bar will disappear.

 

Similarly, if resizing the window causes the page to become too long to fit in the window, the vertical scroll bar will be activated. If the page becomes shorter than the window, the vertical scroll bar will become greyed out.

 

1.5.4    Rendered Elements

BrowserMonkey will not render any content enclosed by the <head> </head> tag pair.

Text not within a heading or font tag pair will be rendered as 12pt Times New Roman. Rendered text will not be selectable by the user.


2      Requirement 2 - Logging

Ability to record the details of non-conformant HTML pages and broken links in a log file.

2.1    Activating logging

The logging feature will be activated by a command line switch when starting the program:

 

java BrowserMonkey -l

2.2    Log file name

The log file will be named “log.txt” and will reside in the current directory. If “log.txt” does not exist BrowserMonkey will create it. If it does exist, any new entries will be appended to the existing ones.

2.3    Log file size

To avoid problems caused by an expanding log file in a device with possibly very limited filespace, BrowserMonkey will not activate logging if the log file has already reached or exceeded an arbitrary maximum size. If the log file approaches the maximum size whilst the program is running, BrowserMonkey will simply deactivate the logging.

 

The maximum size will be set to 45K which will allow for approximately 658 messages (assuming an average message size of 70 characters).

2.4    Log file entry format

Log file entries will be 1 entry per line in the format: date time file linenumber error

 

31/12/2009 23:58 C:\Manuals\thresher.html Expected ‘>’ found ‘#’

 

date                   dd/mm/yyyy            e.g. 31/12/2009

time                   hh:mm                     e.g. 23:58

file                      path/name.extn     e.g. C:\Manuals\thresher.html

error                                                    e.g. Expected ‘>’ found ‘#’

linenumber       on line n                 e.g. on line 932

 

 

If the log file is to be viewed using a browser rather than a text editor, its entire contents should first be enclosed in a <pre> </pre> tag pair to allow symbols such as < to display correctly. It is anticipated that this action would be carried out by the HTML writers where necessary.

3      Requirement 3 – Conformance

3.1    Overview

The browser is to indicate whether the HTML page conforms to the HTML specification. For the purposes of this project, the HTML specification is assumed to be the HTML 4.01 transitional specification. Successfully rendered conformant pages will display the status bar message:

 

“Loaded. Page appears to conform to the HTML specification”

 

Pages considered to be non-conformant will display the message

 

“Loaded. Page may not fully conform to the HTML specification”

 

and where possible will give details of the line number and non-conforming element.

 

BrowserMonkey can only give an indication of whether a page conforms to the HTML specification. A complete and fully accurate assessment of a page’s conformance would require the page to be compared against the HTML 4.01 transitional Document Type Definition (DTD), which is beyond the scope of this project.

 

In its notes on invalid documents2 the Worldwide Web Consortium provides the following guidance:

 

This specification does not define how conforming user agents handle general error conditions, including how user agents behave when they encounter elements, attributes, attribute values, or entities not specified in this document.

However, to facilitate experimentation and interoperability between implementations of various versions of HTML, we recommend the following behavior:

·         If a user agent encounters an element it does not recognize, it should try to render the element's content.

·         If a user agent encounters an attribute it does not recognize, it should ignore the entire attribute specification (i.e., the attribute and its value).

·         If a user agent encounters an attribute value it doesn't recognize, it should use the default attribute value.

·         If it encounters an undeclared entity, the entity should be treated as character data.

We also recommend that user agents provide support for notifying the user of such errors.

 

This section describes the conformance checks which BrowserMonkey will perform, and the action it will take when it encounters non-conforming elements.

Text enclosed between the <a> and </a> will be rendered as blue underlined clickable link text. The blue used will be the blue from the Windows VGA palette, ie. #0000FF. BrowserMonkey will not distinguish between unvisited, visited and active links.

 

Closing tag

</a>

If the closing tag is omitted, the remainder of the page will be rendered as though the </a> tag exists at the very end of document.

 

Attributes

<a href="C:\DataFiles\Manuals\thresher.html">

<a href="http://www.customersite.com/thresher.html">

 

The href attribute must contain the path, filename and extension of the target page. The specified target file may be held locally or accessed via the internet. If the file is not found, the error page will be displayed.

 

The path, filename and extension (including “http://” if entered) are limited to a maximum of 255 characters.

 

Any other attributes within the tag brackets will be ignored.

  • <a> is considered to be an inline element, so only other inline elements can nest within <a>  </a>.

 

 

 

7      Requirement 7 – Search

Description

BrowserMonkey will provide the ability to search the rendered text for a specified string. The rendered page can be searched for any given string entered in the search textbox. Having typed the string into the text box, the user will click the search button (or press the enter key) and the page will be re-rendered with every instance of the search string highlighted. The highlight will be comprised of a light green background (#38D878) and white (#FFFFFF) text regardless of the originally rendered colors. The search will be case insensitive, and search only rendered text not tags or attributes.

 

If no occurrences of the search string are found in the rendered text, a “No occurrences of string found” message will be displayed in the status bar.

 

 

8      Requirement 8 – Zoom

Description

BrowserMonkey will provide the ability to increase and decrease the size of the rendered text. Two buttons on the GUI will be used to zoom the text in and out.

 

The first time the zoom in button is pressed the rendered elements will increase in size by 120%. The second time it is pressed they will increase by 150% from the original size, and the third time by 200%. When the rendered elements have been zoomed in by the maximum allowed (200%) then the zoom in button will be greyed out to let the user know that no more zoom in operations can be performed.

 

The first time the zoom out button is pressed the rendered elements will decrease by 50% and then 75% from the original size. When the rendered elements have been zoomed out by the maximum allowed (75%) then the zoom out button will be greyed out to let the user know that no more zoom out operations can be performed.

9      Analysis Model

9.1    Stakeholder and Actor Descriptions

There are three types of actors interacting with the system:

 

  1. Low-Tech Operator
  2. High-Tech Operator
  3. HTML writer

 

and one type of stakeholder:

 

1.   Developer

 

9.1.1    Low-Tech Operator:

The low-tech operator is a farmer who uses the system to obtain information regarding his occupation. He is provided with a touch screen device which enables him to open an HTML page and navigate through it. For a farmer this can be really useful as they can view the manual for the machine they are operating. A farmer can zoom in or zoom out the page for better recognition of desired elements on the page. This can be really helpful especially for people who have short-sightedness. Furthermore, the farmer can follow links to find more specific information.

 

9.1.2    High-Tech Operator:

The high-tech operator is again a farmer who uses the system but he is provided with an extensive tool -a keyboard. This enables him, in addition to all the actions the low-tech operator can perform, to use the keyboard for searching for particular text in the page. This is really useful for a farmer as he might not have the time to read the whole page for finding some information and he can just type text and the system will search to find it.

 

9.1.3    Html writer:

Users of the system of this type are HTML writers who know the full syntax of HTML and can easily develop a web page using the tag notation of HTML. These are the people who create the pages according to the company’s requirements and of course the use of HTML. Their interaction with the system is mainly to write the HTML files and update them for the system.

 

 

9.1.4    Developers:

Developers are the people who have developed the system and can at any time in the future change the program according to new requirements. They can update the system, modify it or add some more features in it if, for example, in the future the company decides to enable users to write their comments. This will definitely need developers’ involvement for modifying the system in a way to allow the rendering of more HTML tags for text input (example for rendering forms). Of course developers are needed for fixing a problem when there is something wrong with the system. Developers just like HTML writers do not interact directly with the system they are just stakeholders and they use another environment to modify the system.

 

 

 

9.2    Use Cases

 

 

 

 

 

9.2.1    Basic and Alternatives course of events:

 

  1. View manual: Basic course of events:
    1. User clicks open to open a file
    2. System displays a browsing area for the user to choose among files.
    3. User browses the files directories to find the one he wants to open
    4. The system displays the HTML page successfully.
    5. The user views the manual

 

1. View manual: Alternative course of events:

    1. User clicks open to open a file
    2. System displays a browsing area for the user to choose among files.
    3. User browses the files directories to find the one he wants to open
    4. The system displays the HTML page with missing content because of wrong use of the tags or displays a pop up window because the directory does not exists.

 

  1. Zoom page: Basic course of events:
    1. The user performs the “View manual” use case.
    2. The user choose zoom in or zoom out as many times as he needs.
    3. The system displays the zoomed in or out page.

 

2. Zoom page: Alternative course of events:

1. The user performs the “View manual” use case.

2. The user choose zoom in or zoom out as many times as he needs.

3. The system displays the same page because zoom in or out has exceeded its limits.

 

  1. Search page: Basic course of events:
    1. The user performs the “View manual” use case.
    2. The user types a text to be searched in the page.
    3. The system displays the searched text in the page by highlighting all of the occurrences of the searched text.

 

3. Search page: Alternative course of events:

    1. The user performs the “View manual” use case.

2. The user types a text to be searched in the page.

3. The system displays a pop up window telling the user that the text have not been found

 

  1. Follow link: Basic course of events:
    1. The user performs the “View manual” use case.
    2. The user presses on a link.
    3. The system displays the linked page.

 

4. Follow link: Alternative course of events:

    1. The user performs the “View manual” use case.
    2. The user presses on a link.
    3. The system is not able to find the page and displays an error informing the user.
